## 推断

- 这个项目不应被命名为单门商业逻辑课 wiki，因为商业逻辑只是第一批课程资料。
- 这个项目更适合作为“高玮课程知识库”，课程是来源组织维度，wiki 是知识组织维度，applications 是使用方式。
- 不建议用 git 分支区分学习内化、商业分析工具箱、AI 知识底座、个人第二大脑四个目标。它们更适合作为同一知识底座上的不同视图。
- 原始资料目录和正式 wiki 项目分开，有利于后续迁移、自动化、版本控制和大文件治理。
- 第一版不需要做完整自动化，但需要先固定可扩展边界：来源层、课程层、知识层、应用层和维护层。

## 待确认问题

- 是否长期采用 `gao-course-wiki` 作为项目名，还是后续改为更中文或更知识体系化的名称。
- 视频和 PDF 原件未来是否进入单独资料仓库，还是继续留在 HomeSpace。
- Obsidian 是否作为第一展示软件，还是只要求 Markdown 通用兼容，未来再接 Quartz、MkDocs 或 RAG。
- 商业逻辑课第一批摄入时，是先处理全部文字稿，还是先选 5 到 10 篇样本跑通流程。
- 高玮课程内容与个人观点、外部资料、公司经验之间是否要做明确来源隔离。

## 设计约束

- 项目文档不能出现 emoji。
- 调研结论必须区分事实、推断和待验证问题。
- 涉及外部资料时，优先保留来源链接或本地参考路径。
- 课程资料如果涉及版权或私有访问权限，wiki 中应保留摘要、索引和引用，不默认公开发布全文。
- Agent 生成的综合结论必须能追溯到来源讲次、来源片段或人工确认记录。
