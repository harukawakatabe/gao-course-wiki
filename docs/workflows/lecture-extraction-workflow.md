# 讲次提炼工作流

本文定义高玮课程 LLM Wiki 的讲次提炼闭环。目标是让 agent 每处理一回讲次，都完整更新来源层、课程层、知识层、入口索引、日志和 git 记录，并在批量连续提炼时保留必要检查点。

## 核心原则

- 一回讲次就是一个最小闭环，不能只更新讲次页而跳过来源卡片、wiki 页面、索引、日志或校验。
- 讲次编号必须锁定原始 `LectureText` 目录中的三位编号。当前原始 txt 已经过人工校正，常规提炼默认信任已登记的 `source_path` 和来源卡片标题；只有发现编号、标题或正文主题异常时，才进入异常复核并记录差异，不能静默改写。
- 事实、推断、待验证问题必须分开写。agent 综合出来的稳定结论必须能追溯到来源卡片和讲次页。
- 讲次页保留原课程逻辑顺序，wiki 页只沉淀可跨讲次复用的概念、框架、案例或方法。
- 文档不能出现 emoji；未经人工确认的页面保持 `status: draft` 和 `human_checked: false`。
- 每次重要提炼都要更新 `log.md`，通过 lint 后提交 git。

## 使用声明

agent 因触发口令或主动判断而调用本文执行单篇或批量提炼时，必须在开始执行前显式声明正在使用 `docs/workflows/lecture-extraction-workflow.md` 讲次提炼工作流。

## 单篇提炼工作流

### 单篇提炼入口

触发口令：

```text
提炼讲次 042
```

agent 收到单篇提炼任务后，按以下阶段执行。

### 阶段一：工作区和来源预检

1. 运行 `git status --short`，确认是否存在未提交改动。
2. 若存在未提交改动，先判断是否与本次讲次相关：
   - 相关：读取差异，继续在现有改动基础上工作。
   - 不相关：不要覆盖，不要回滚，只在后续提交中避免加入这些文件。
3. 读取来源卡片 `sources/cards/source-business-logic-<id>.md`。
4. 从来源卡片取得 `source_path`，确认原始 txt 文件存在。
5. 默认不再逐篇强制核对文件名标题和正文开场主题；仅在以下情况触发异常复核：
   - 来源卡片讲次编号、目标讲次编号和 `source_path` 中的三位编号不一致。
   - 来源卡片标题、已有讲次页标题或原文提炼出的主题互相冲突。
   - 来源卡片或讲次页已有错位、校正、待核验等说明。
   - 阅读原文时发现正文内容明显不像目标讲次。
6. 触发异常复核时，再读取原始 txt 的文件名、正文开场标注和必要上下文，并在来源卡片和讲次页中保留“来源说明”或“来源核验”。

本阶段的硬性停止条件：

- `source_path` 不存在。
- 来源卡片讲次编号与目标讲次编号不一致。
- 触发异常复核后，仍无法确认正文主题或正确来源。
- 发现已有页面内容明显来自另一个讲次，但无法确定正确来源。

停止时只在回复中说明问题，不继续提炼。

### 阶段二：提炼原文

阅读全文后先在工作记忆中提取以下内容，再开始写文件：

- 本讲定位：本讲解决什么问题，放在课程体系中的位置是什么。
- 核心问题：3 到 6 个问题，优先保留老师的原始问题意识。
- 事实：原文明确表达的观点、框架、案例、步骤和判断。
- 推断：基于原文可以合理推出、但不是原文直接说出的综合判断。
- 待验证问题：需要后续讲次、外部资料或人工判断校正的内容。
- 可进入 wiki 的稳定知识：概念、框架、案例、行业判断、组织问题。
- 不进入 wiki 的单讲内容：只在本讲成立的案例细节、过渡表达和课堂语境。

提炼时不得逐字复制大段原文。必要短句可以保留，但正文应以自己的语言重写。

### 阶段三：更新讲次页

目标文件位于：

```text
courses/business-logic/lectures/
```

讲次页必须更新以下部分：

- frontmatter 的 `updated` 日期。
- 标题与正文主题，默认沿用已校正来源卡片和原文主题。
- `## 来源说明`；若触发异常复核或历史上存在文件名和正文主题错位，必须保留 `## 来源核验` 或等价说明。
- `## 本讲定位`
- `## 核心问题`
- `## 主要观点`
- `## 概念和框架`
- `## 案例`
- `## 可复用启发`
- `## 待验证问题`
- `## 关联`

讲次页写作要求：

- 保留课程原始展开顺序。
- 事实和推断分小节，避免混写。
- 案例只写与本讲框架有关的部分。
- `## 关联` 至少包含来源卡片和相关 wiki 页面。
- 页面中不能留下“待从来源文字稿中提炼”，除非本次明确只是创建骨架。

### 阶段四：更新来源卡片

目标文件：

```text
sources/cards/source-business-logic-<id>.md
```

来源卡片必须更新：

- `title`：默认以已校正正文主题或来源卡片标题为准；若异常复核发现与文件名不同，在正文说明。
- `source_path`：严格保留锁定编号的原始路径。
- `status`：完成提炼后改为 `summarized`。
- `updated` 日期。
- `## 来源说明`：记录来源性质；常规情况下说明文本已校正并锁定路径，异常复核时记录任何编号、文件名、正文主题错位。
- `## 文件信息`：路径、格式、课程、讲次。
- `## 已处理产物`：讲次页和已更新 wiki 页。
- `## 事实摘录`：只摘录原文明确内容，不写二次发挥。
- `## 推断和待验证`：记录综合推断和后续需要校正的问题。
- `## 关联页面`：列出讲次页和 wiki 页。

来源卡片不是讲次摘要，事实摘录要精炼，重点服务可追溯性。

### 阶段五：更新 wiki 页面

根据本讲内容选择一种处理方式：

- 更新已有 wiki 页：例如 `wiki/frameworks/review-method.md`。
- 新建 wiki 页：当本讲产生新的稳定概念、框架、案例或行业问题。
- 暂不进入 wiki：当内容还只是单讲案例，尚未形成稳定知识。

wiki 页面必须满足：

- frontmatter 有 `source_scope`，包含本讲 `course` 和 `lecture_id`。
- `human_checked: false`，除非用户明确人工确认。
- 正文区分定义、适用条件、操作步骤、典型案例、易错点、来源和证据。
- `## 来源和证据` 至少链接来源卡片或讲次页。
- 单讲专有内容不强行上升为跨讲次结论。
- 若更新已有页面，要追加来源范围，不能删除已有讲次证据。

### 阶段六：更新索引和 AI 入口

每篇提炼完成后检查以下入口是否需要更新：

- `courses/business-logic/index.md`：新增讲次或标题变化时更新。
- `wiki/index.md`：新增 wiki 页或标题变化时更新。
- `applications/ai-context/index.md`：当新增重要入口、工具页或索引规则变化时更新。
- `sources/index.md`：通常不需要每篇更新，除非新增来源状态说明。
- `sources/raw-paths.md`：只有原始资料目录变化或重新盘点时，运行 `python scripts/inventory_sources.py` 更新。

索引更新原则：

- 已存在条目只在标题或路径需要修正时改。
- 新增 wiki 页必须进入 `wiki/index.md`。
- 不为了单篇提炼反复改动无关入口。

### 阶段七：校验

每篇提炼后必须运行：

```powershell
python scripts\lint_wiki.py
```

若项目环境使用 uv，也可以运行：

```powershell
uv run python scripts\lint_wiki.py
```

校验通过后，再做人工式检查：

- 目标讲次页没有残留占位文本。
- 来源卡片 `status` 已改为 `summarized`。
- 来源路径、讲次编号、讲次页文件名、wiki `source_scope` 一致。
- 若存在文件名和正文主题错位、校正或异常复核记录，说明没有被删除。
- wiki 页面没有把推断写成事实。
- `log.md` 准备记录本次处理。

### 阶段八：更新日志

在 `log.md` 追加一条记录，格式：

```markdown
## YYYY-MM-DD | lecture-<id>-summarized | 提炼讲次 <id>

- 读取来源卡片 `sources/cards/source-business-logic-<id>.md`，确认文字稿路径为 `<source_path>`。
- 来源确认：<常规确认已校正来源 / 异常复核结果>。
- 更新讲次页面 `<lecture_path>`，覆盖本讲定位、核心问题、主要观点、概念框架、案例和可复用启发。
- 更新来源卡片状态为 `summarized`，补充事实摘录和推断/待验证项。
- 更新 wiki 页面 `<wiki_path>`，沉淀 <概念或框架名称>。
- 运行 `python scripts\lint_wiki.py`，结果通过。
```

若本讲只更新讲次页和来源卡片，未更新 wiki，日志必须写明原因。

### 阶段九：提交 git

提交前运行：

```powershell
git status --short
git diff --stat
```

只暂存本次提炼相关文件。不要把本次之前已经存在、且与本讲无关的改动混入提交。

推荐提交命令：

```powershell
git add <本次相关文件>
git commit -m "docs: summarize business logic lecture <id>"
```

如果本次是修复来源绑定或重新提炼，使用：

```powershell
git commit -m "docs: correct business logic lecture <id> source extraction"
```

提交后再次运行：

```powershell
git status --short
```

确认只剩无关改动，或工作区干净。

## 批量提炼工作流

### 批量提炼入口

触发口令：

```text
批量提炼 001 020 034 042 044
```

批量提炼不是把所有讲次一次性写完，而是按单篇闭环循环执行。

### 批量执行节奏

推荐节奏：

1. 批量前预检全部目标讲次来源卡片和 `source_path`。
2. 生成一个批量队列，记录每个讲次的状态：待处理、处理中、已提交、阻塞。
3. 每篇讲次完整执行单篇九阶段流程。
4. 每篇讲次通过 lint 后单独提交 git。
5. 每处理 3 到 5 篇，做一次批量检查点。
6. 批量结束后做总检查和总结。

不推荐一次性处理完多篇后再统一提交。原因是来源错位、wiki 合并和日志冲突会放大返工成本。

### 批量前预检

批量开始前逐篇检查：

- 来源卡片是否存在。
- `source_path` 是否存在。
- 原始 txt 文件编号是否与目标编号一致。
- 是否存在错位、校正、待核验等历史说明。
- 对应讲次页是否存在。
- 对应 wiki 页是否存在或是否需要新建。

批量预检默认不逐篇阅读全文或核对正文开场主题；只有发现编号不一致、历史错位说明或标题冲突时，才触发异常复核。预检结果建议写入工作回复，不必写入仓库，除非发现结构性问题。

若任一讲次存在阻塞：

- 将该讲次标记为阻塞。
- 继续处理其他不受影响的讲次。
- 批量结束时集中汇报阻塞原因。

### 批量检查点

每处理 3 到 5 篇，暂停做检查点：

```powershell
python scripts\lint_wiki.py
git status --short
git log --oneline -5
```

同时人工式检查：

- 是否出现多个 wiki 页表达同一概念。
- 是否把某一讲案例过度抽象成稳定框架。
- 是否有来源错位说明被简化或删除。
- `log.md` 记录是否能还原每篇处理过程。
- wiki `source_scope` 是否累积多个讲次而不是被覆盖。

检查点通过后继续批量；不通过则先修正，再继续。

### 批量结束检查

批量结束后运行：

```powershell
python scripts\lint_wiki.py
git status --short
git log --oneline -10
```

最终检查：

- 所有目标讲次来源卡片状态符合实际处理状态。
- 所有已完成讲次页无占位文本。
- 新增或更新的 wiki 页都能追溯到来源。
- `courses/business-logic/index.md` 和 `wiki/index.md` 已覆盖新增页面。
- `log.md` 每篇都有记录。
- 每篇或每组相关修正都有清晰 git commit。

## agent 执行提示模板

### 单篇执行提示

可直接把以下提示交给 agent：

```text
请按 `docs/workflows/lecture-extraction-workflow.md` 执行。
目标：提炼讲次 <id>。
要求：
1. 开始前声明正在使用讲次提炼工作流。
2. 先核对 git 状态和来源卡片。
3. 锁定来源卡片和 `source_path`；默认信任已校正文本，只有发现编号、标题或正文主题异常时才复核并记录。
4. 完整更新讲次页、来源卡片、必要 wiki 页、索引和 log.md。
5. 运行 `python scripts\lint_wiki.py`。
6. 只提交本讲相关文件，提交信息使用 `docs: summarize business logic lecture <id>`。
7. 最后汇报更新文件、lint 结果、commit hash 和待验证问题。
```

### 批量执行提示

```text
请按 `docs/workflows/lecture-extraction-workflow.md` 执行批量提炼。
目标讲次：<id 列表>。
要求：
1. 开始前声明正在使用批量讲次提炼工作流。
2. 批量前先做来源预检，列出每篇状态。
3. 每篇按单篇闭环执行并单独提交。
4. 每 3 到 5 篇做一次检查点。
5. 遇到阻塞讲次先记录并跳过，不影响其他讲次。
6. 批量结束后运行 lint，汇报完成讲次、阻塞讲次、commit 列表和需要人工复核的问题。
```
